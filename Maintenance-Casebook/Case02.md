# L3 보안 서버 이중 연결 환경에서 MSTP로 인한 통신 장애

## 작업 개요

기존 스위치 A와 스위치 B 사이에 L3 기능을 제공하는 보안 서버 2대를 이중화 형태로 구축하였다.

각 보안 서버는 각각 스위치 A와 스위치 B에 연결되도록 구성하여 이중 경로를 확보하고자 하였으며, 보안 서버에서는 VLAN과 서로 다른 서브넷 간 라우팅을 수행하도록 설계하였다.

논리적으로는 L3 라우팅 구조이므로 Layer 2 루프는 발생하지 않을 것으로 예상하였다.

---

## 네트워크 구성

```text
          +-------------+
          |  Security A |
          +-------------+
         /               \
    Switch A           Switch B
         \               /
          +-------------+
          |  Security B |
          +-------------+
```

- Switch A ↔ Security A
- Security A ↔ Switch B
- Switch A ↔ Security B
- Security B ↔ Switch B

보안 서버에서 VLAN 간 라우팅을 수행하는 구조

---

## 증상

- 보안 서버 1대만 연결하면 정상 통신
- 보안 서버 2대를 모두 연결하면 Ping 통신 실패
- 한쪽 링크를 분리하면 정상 통신
- VLAN과 서브넷은 서로 다르게 구성되어 있었음

---

## 원인 분석

초기에는 VLAN과 서브넷이 서로 다르므로 Layer 2 Loop가 발생하지 않을 것으로 판단하였다.

그러나 스위치의 MSTP(Multiple Spanning Tree Protocol) 상태를 확인한 결과, 스위치가 물리적인 연결 구조를 루프 토폴로지로 인식하여 일부 포트를 Blocking 상태로 전환하고 있었다.

즉,

- L3에서는 서로 다른 서브넷
- 물리적인 Layer 2 연결은 루프 형태

였기 때문에 MSTP가 루프 방지를 위해 포트를 차단하고 있었으며, 이로 인해 정상적인 통신이 이루어지지 않았다.

---

## 확인 절차

1. Ping 테스트
2. VLAN 및 IP 설정 확인
3. 라우팅 설정 확인
4. 링크 상태 확인
5. MSTP 상태 확인
6. Blocking Port 확인

---

## 조치 내용

해당 환경은 보안 서버에서 Layer 3 라우팅을 수행하는 구조였으며, 운영 정책에 따라 스위치의 MSTP를 비활성화하여 포트 차단이 발생하지 않도록 조치하였다.

조치 후 모든 링크가 정상적으로 Forwarding 상태가 되었으며 통신이 정상적으로 이루어졌다.

> **참고**
>
> STP 비활성화는 일반적인 Layer 2 네트워크에서는 루프를 유발할 수 있으므로 신중하게 적용해야 한다. 본 사례는 L3 기반 설계와 운영 환경을 충분히 검토한 후 적용한 사례이다.

## 배운 점

- 서로 다른 VLAN과 서브넷을 사용하더라도 물리적인 연결 구조는 STP의 영향을 받을 수 있다.
- STP는 IP나 라우팅 정보를 기반으로 동작하는 것이 아니라 Layer 2 토폴로지를 기준으로 루프를 판단한다.
- 장애 분석 시에는 Layer 3뿐 아니라 Layer 2 프로토콜(STP/MSTP/RSTP)의 동작 여부도 함께 확인해야 한다.

---

## 핵심 원인

**서로 다른 VLAN과 서브넷을 사용하더라도 물리적인 연결 구조가 루프 형태였기 때문에 MSTP가 이를 감지하여 포트를 Blocking 상태로 전환했고, 이로 인해 통신 장애가 발생하였다.**
