# Case 07. 서버 NIC Bonding 이상으로 인한 서버 통신 장애

## 작업 개요

방화벽 - L2 스위치 - 서버 구조의 네트워크 환경에서 서버로 통신이 되지 않는 장애가 발생하였다.

장애 접수 후 현장에 방문하여 네트워크 장비를 점검하고, 방화벽 및 서버 담당자와 협업하여 원인을 분석하였다.

---

## 네트워크 구성

```text
Client
   │
Firewall
   │
L2 Switch
   │
Server (NIC Bonding)
```

---

## 증상

- 서버로 Ping 통신 불가
- 서버 서비스 접속 불가
- L2 스위치에는 특이사항 없음
- 서버를 제외한 네트워크 구간까지는 Ping 정상

---

## 원인 분석

먼저 L2 스위치의 포트 상태, VLAN 설정 및 인터페이스 상태를 확인하였으나 이상은 발견되지 않았다.

L2 스위치는 단순 스위칭 장비이므로 서버를 제외한 구간까지 Ping 테스트를 수행한 결과 정상적으로 통신되는 것을 확인하였다.

이후 방화벽 엔지니어가 `tcpdump`를 이용하여 패킷 흐름을 확인한 결과,

- 방화벽까지 패킷 정상 수신
- 방화벽에서 서버 방향으로 패킷 정상 전달
- 서버와의 통신만 실패

하는 것을 확인하였다.

이를 통해 방화벽 및 네트워크 구간보다는 서버 측에 문제가 있을 가능성이 높다고 판단하였다.

서버 담당자가 서버를 점검한 결과, NIC에 정상적인 네트워크 정보가 적용되어 있지 않았으며 NIC Bonding도 정상적으로 동작하지 않는 상태를 확인하였다.

---

## 확인 절차

1. L2 스위치 포트 상태 확인
2. VLAN 및 인터페이스 설정 확인
3. Ping 테스트를 통한 장애 구간 확인
4. 방화벽 엔지니어의 `tcpdump` 결과 확인
5. 서버 담당자의 NIC 및 Bonding 상태 확인

---

## 조치 내용

서버 담당자가 다음 작업을 수행하였다.

- NIC 네트워크 정보 재설정
- NIC Bonding 재구성

조치 후 서버와의 통신이 정상적으로 복구되었다.

---

## 결과

- 서버 Ping 정상
- 서비스 정상 복구
- 추가적인 통신 이상 없음

---

## 원인 추정

장애 발생 당일 오전 스위치 재부팅 작업이 수행된 이력이 있었다.

정확한 원인은 확인되지 않았지만 다음과 같은 가능성을 추정하였다.

- 스위치 재기동 이후 서버의 NIC Bonding이 정상적으로 복구되지 않았을 가능성
- 기존 Bonding 설정 자체에 문제가 있었을 가능성

다만 관련 로그가 남아 있지 않아 근본 원인(Root Cause)은 확인하지 못하였다.

---

## 배운 점

- 네트워크 장애 분석 시 장애 구간을 단계적으로 좁혀 나가는 것이 중요하다.
- L2 스위치, 방화벽, 서버를 순차적으로 확인하여 문제 발생 지점을 빠르게 특정할 수 있었다.
- 방화벽의 패킷 분석 결과와 서버 담당자의 점검 결과를 종합하여 원인을 파악하였다.
- NIC Bonding 환경에서는 스위치 작업 이후 Bonding 상태도 함께 확인하는 것이 중요하다.
- 확인된 사실과 추정 사항을 명확히 구분하여 기록하는 것이 정확한 장애 분석에 도움이 된다.

---

## 핵심 원인

**네트워크 장비에는 이상이 없었으며, 서버의 NIC 네트워크 설정 및 Bonding이 정상적으로 동작하지 않아 서버와의 통신이 실패하였다. 서버 측에서 NIC 설정 및 Bonding을 재구성한 후 정상적으로 통신이 복구되었다.**

> **참고:** 스위치 재부팅과 Bonding 이상 사이의 연관성은 추정되었으나, 관련 로그가 없어 직접적인 원인으로 확정할 수는 없었다.
