# Case 05. 방화벽 정책 변경으로 인한 단말 통신 장애

## 작업 개요

건물 A와 서버실 간 광 연결 환경에서 며칠 전까지 정상적으로 운영되던 네트워크가 갑자기 통신되지 않는다는 장애가 접수되었다.

특이한 점은 서버 간 통신은 정상적으로 이루어졌지만, PC와 같은 사용자 단말은 통신이 되지 않는 현상이 발생하였다.

---

## 네트워크 구성

```text
PC
 │
 │
건물 A Switch
 │
 │ Fiber
 │
서버실 Switch
 │
 │
방화벽(Watching Mode)
 │
 │
Server
```

---

## 증상

- 서버 간 통신 정상
- PC 등 사용자 단말 통신 실패
- Gateway Ping 실패
- 장애 발생 전까지 정상 운영
- 물리 링크 및 광 연결 정상

---

## 원인 분석

먼저 물리적인 연결 상태와 네트워크 설정을 확인하였으나 이상은 발견되지 않았다.

이후 단말의 ARP Table을 확인한 결과 Gateway MAC Address가 정상적으로 학습되지 않는 현상을 확인하였다.

당시에는 ARP가 학습되지 않는 원인을 즉시 특정하지 못하여 최근 변경 사항을 확인하기 위해 담당자에게 문의하였다.

확인 결과, 며칠 전 보안 감사에 대비하여 방화벽의 Watching Mode 정책이 기존보다 강화된 상태로 변경되어 있었다.

방화벽 정책을 기존 운영 정책으로 복원한 후 ARP 학습이 정상적으로 이루어졌으며, 단말 통신도 함께 정상 복구되었다.

---

## 확인 절차

1. 물리 링크 상태 확인
2. Ping 테스트
3. IP 및 Gateway 확인
4. ARP Table 확인
5. 최근 변경 사항 확인
6. 방화벽 정책 확인
7. 방화벽 정책 원복
8. 통신 테스트

---

## 조치 내용

- 최근 변경 사항 확인
- 방화벽 정책(Watching Mode) 원복
- ARP 학습 여부 확인
- 단말 통신 확인

---

## 결과

- ARP Table 정상 학습
- Gateway 통신 정상
- PC 통신 정상
- 서비스 정상 복구

---

## 배운 점

- 장애 분석 시 장비 설정뿐만 아니라 **최근 변경 이력(Change History)** 을 확인하는 것이 매우 중요하다.
- 서버는 정상인데 사용자 단말만 통신되지 않는 경우에는 보안 장비의 정책 변경 여부도 함께 점검해야 한다.
- ARP 학습이 이루어지지 않는 현상이 발생할 경우 네트워크 장비뿐 아니라 방화벽, NAC 등 보안 장비의 정책도 원인 후보가 될 수 있다.

---

## 핵심 원인

**보안 감사를 위해 강화된 방화벽 정책으로 인해 단말의 정상적인 통신이 이루어지지 않았으며, 정책을 기존 운영 상태로 복원한 후 ARP 학습과 통신이 정상적으로 복구되었다.**

> **참고:** 당시에는 ARP 학습이 되지 않는 현상을 먼저 확인하였으며, 원인을 추적하는 과정에서 최근 변경된 방화벽 정책을 확인하여 장애를 해결하였다.
